# UI_SPEC.md — THE HYUNDAI POS (source-based, current UI)

> 이 문서는 **현재 업로드된 소스 코드 기준**으로 정리한 UI 스펙이다.  
> 대상 소스:
> - `ui/component/PosTopBar.kt`
> - `ui/screen/SuperHomeScreen.kt`
> - `ui/screen/RestaurantScreen.kt`
> - `ui/screen/FoodCourtScreen.kt`
> - `ui/screen/ProductRegisterScreen.kt`
> - `ui/screen/PaymentScreen.kt`
> - `navigation/MainNavHost.kt`
> - `ui/theme/Theme.kt`
>
> 목적:
> - 현재 구현된 UI 컨셉과 화면 구조를 문서화한다.
> - 이후 기능 추가 / 유지보수 / 리팩토링 시 **현재 UI 언어를 보존**하도록 한다.
> - 과거 문서와 다르더라도, 본 문서는 **현재 소스 상태를 우선 반영**한다.

---

## 0) 문서 역할

### 0.1 우선순위
- 이 문서는 **현재 Compose UI 구조 / 배치 / 버튼 라벨 / 고정 영역 / 시각 언어**를 정의한다.
- 아키텍처, 레거시 연동, 비즈니스 로직, 상태 소유권은 `PROMPT.md`를 따른다.
- 문서와 실제 소스가 다를 경우, **현재 소스 구현을 우선 사실로 간주**한다.
- 단, 의미 해석이 애매하면 기존 POS 흐름을 깨지 않는 최소 변경안을 선택한다.

### 0.2 화면 범위
현재 UI 스펙 범위:
- `SUPER_HOME`
- `RESTAURANT`
- `FOOD`
- `PRODUCT_REGISTER`
- `PAYMENT`

### 0.3 실제 내비게이션 경로
현재 소스 기준 실제 route는 아래와 같다.

- `super_home`
- `restaurant`
- `food/{tableId}`
- `product_register?barcode={barcode}`
- `payment/{tableId}`

의미상 대응은 아래와 같다.

- `SUPER_HOME` → `super_home`
- `RESTAURANT` → `restaurant`
- `FOOD` → `food/{tableId}`
- `PRODUCT_REGISTER` → `product_register?barcode={barcode}`
- `PAYMENT` → `payment/{tableId}`

### 0.4 중요한 의미 정의
- `PRODUCT_REGISTER` 는 **상품 마스터 등록 폼이 아니라 판매 등록/판매 입력 화면**이다.
- `FOOD` 는 레스토랑/푸드코트 주문 입력 화면이다.
- `PAYMENT` 는 주문 금액에 대해 수단별 금액을 배분 입력하는 **전용 결제 화면**이다.

## 1) 공통 UI 언어

### 1.1 전체 톤
현재 UI는 다음 톤을 공통으로 사용한다.

- 밝은 베이지/아이보리 기반 페이지 배경
- 흰색 카드(surface) 위주 구성
- 딥그린 강조
- 웜 베이지/브라운 보조 강조
- 슬레이트/블루그레이 보조 버튼
- POS답게 **조밀하지만 읽기 쉬운 정보 밀도**
- 터치 친화적인 큰 버튼 / 큰 숫자 / 명확한 구역 분리

### 1.2 공통 색상 규칙
소스에서 반복적으로 쓰이는 대표 색상은 아래 범위를 기준으로 본다.

- `PRIMARY_GREEN`: `#005645`
- `PREMIUM_GREEN`: `#003D32`
- `ACCENT_BEIGE`: `#C1A57A`
- `ACCENT_SLATE`: `#5A6B7A`
- `PAGE_BG_MAIN`: `#F6F2E9`
- `PAGE_BG_HOME`: `#F1F3F4`
- `SURFACE_BORDER`: `#DADCE0`
- `SOFT_SURFACE`: `#F1F3F4`
- `DANGER_RED`: `#D32F2F`
- `GOLD_ACCENT`: `#FFD54F`

### 1.3 공통 레이아웃 토큰
현재 구현 기준 권장 토큰:

- 화면 외곽 padding: `12 ~ 16dp`
- pane 간 간격: `10 ~ 16dp`
- 카드/패널 radius: `10 ~ 12dp`
- 강조 카드 radius: `14 ~ 16dp`
- 모달/키패드 radius: `24dp`
- 최소 터치 타깃: `48dp`
- 일반 액션 버튼 높이:
  - 중형: `56 ~ 64dp`
  - 대형: `72 ~ 84dp`
  - Hero tile: `130dp`

### 1.4 공통 시각 구조
- `PosTopBar` 는 모든 주요 화면에서 상단 고정이다.
- 대부분 화면은 **좌/우 2-pane** 구조다.
- 리스트/그리드/요약/액션 구역을 카드 또는 Surface로 분리한다.
- 스크롤 가능한 영역에는 상/하단 바운싱 힌트 아이콘을 사용할 수 있다.
- 주요 합계/받을 금액/총액은 큰 폰트와 강한 대비 색으로 강조한다.

### 1.5 폰트/텍스트 톤
- Compose `Material3` + SansSerif 기반.
- 제목 / 합계 / 결제 관련 텍스트는 `Bold ~ ExtraBold`.
- 표 헤더/보조 정보는 회색 계열.
- 금액은 진한 색 또는 강조색으로 분리한다.

## 2) 공통 상단 탑바 — `PosTopBar`

### 2.1 구성
상단 탑바는 모든 주요 화면 공통이며, 스크롤과 무관하게 고정한다.

왼쪽부터 순서:

1. `THE HYUNDAI` 브랜드 텍스트
2. 날짜/시간 블록
3. 정보 칩 2개
   - `POS번호`
   - `거래번호`
4. 우측 액션 버튼 4개
   - `점검`
   - `조회`
   - `재출력`
   - `더보기`

### 2.2 시각 규칙
- 흰 배경 + 약한 그림자
- 브랜드 텍스트는 딥그린
- 날짜/시간은 2줄 구성
- 정보 칩은 연한 회색 배경
- 우측 액션 버튼은 Filled Tonal Button 스타일
- 액션 버튼 높이는 `48dp` 기준

### 2.3 고정 규칙
- 모든 주요 화면에서 탑바는 항상 노출한다.
- 본문 스크롤과 분리한다.
- 탑바 아래부터 각 화면의 고유 구조가 시작된다.

## 3) SUPER_HOME

### 3.1 목적
- 앱 시작 화면
- 바코드 입력/스캔 진입점
- 판매 등록 진입점
- 식당 관리 진입점
- 최근 거래 요약 확인
- 유틸리티 기능 접근

### 3.2 전체 구조
- 상단: `PosTopBar`
- 본문: 좌/우 2-pane
  - 좌 `65%`
  - 우 `35%`
- 하단: 고정 유틸리티 바

### 3.3 화면 배경
- `PAGE_BG_HOME (#F1F3F4)` 기반
- 본문 카드들은 흰색 Surface + 연한 회색 border

### 3.4 좌측 영역
좌측은 세 덩어리로 구성한다.

#### 3.4.1 바코드 입력 바
- 좌측 최상단
- 긴 흰색 입력 카드
- placeholder:
  - `바코드를 스캔하거나 입력하세요`
- leading icon:
  - 검색 아이콘
- 포커스를 유지하는 입력 UX를 사용한다.
- Enter/개행이 들어오면 `ProductRegister(barcode)` 진입 흐름을 탄다.

#### 3.4.2 Hero Tile Row
가로 3분할 큰 타일:

1. `상품판매`
2. `식당관리`
3. `통합조회`

시각 규칙:
- 높이 약 `130dp`
- 큰 아이콘 + 큰 타이틀 텍스트
- 둥근 모서리 + 그림자
- 색상:
  - `상품판매` = 딥그린
  - `식당관리` = 베이지/브라운
  - `통합조회` = 슬레이트

현재 연결 의미:
- `상품판매` → `PRODUCT_REGISTER()`
- `식당관리` → `RESTAURANT`
- `통합조회` → 현재 placeholder 가능

#### 3.4.3 보조 기능 그리드
- 흰색 카드 내부 4열 그리드
- 각 아이템은 아웃라인 버튼 스타일
- 기본 라벨:
  - `즐겨찾기`
  - `행사보관`
  - `영수증 조회`
  - `소비기한 등록`
  - `상품조회`
  - `포인트 조회`
  - `보류/복원`
  - `도중회수`

현재 구현상 일부는 placeholder일 수 있다.
버튼은 UI상 반드시 유지한다.
현재 소스 기준으로는 `즐겨찾기` 버튼이 임시로 Restaurant 이동에 연결되어 있다.

### 3.5 우측 영역
우측은 2개 블록으로 구성한다.

#### 3.5.1 직전 결제 요약 카드
표시 항목:
- `상품 수량`
- `총 금액`
- `할인 금액`
- `거스름돈`

규칙:
- 상단에 시계 아이콘 + `직전 결제 요약`
- 카드 내부 구분선 사용
- `거스름돈`은 빨간색 강조

#### 3.5.2 세로형 바로가기 버튼
버튼 라벨:
- `직전 영수증 출력`
- `직전 거래 환불`
- `보류 내역`
- `도중 회수`

규칙:
- 전폭 버튼
- 슬레이트 계열 배경
- 높이 약 `64dp`

### 3.6 하단 유틸리티 바
하단 고정 바에 6개 버튼:

- `고정 메뉴`
- `시재 점검`
- `셀프계산대 전환`
- `전달노트`
- `따라하기`
- `직원외 할인`

규칙:
- 탑바와 별개로 화면 최하단 고정
- 회색 배경 위에 동일 비중 버튼 배치
- 각 버튼 높이 약 `56dp`

### 3.7 금지 사항
- SuperHome을 단일 컬럼 모바일 홈처럼 바꾸지 않는다.
- Hero 3타일 구조를 임의로 제거하지 않는다.
- 하단 유틸리티 바를 스크롤 영역에 넣지 않는다.

## 4) RESTAURANT

### 4.1 목적
- 테이블 현황 조회
- 선택 테이블 주문 요약 확인
- 빈 테이블 주문 진입
- 테이블 이동/합석 대상 선택 및 확인

### 4.2 전체 구조
- 상단: `PosTopBar`
- 본문: 좌/우 2-pane
  - 좌: 테이블 영역
  - 우: 고정 주문 패널
- 페이지 배경: `PAGE_BG_MAIN (#F6F2E9)`

### 4.3 좌측 상단 Area 탭
- `ScrollableTabRow`
- area 라벨을 가로 탭으로 표시
- 선택 탭은 딥그린 인디케이터
- 예: 홀/룸/테라스 등 영역 그룹

### 4.4 모드 배너
일반 모드가 아닐 때 좌측 상단에 딥그린 배너 표시.

가능 라벨:
- `이동 대상 테이블 선택`
- `합석 대상 테이블 선택`

### 4.5 테이블 그리드
- 5열 고정 그리드
- 카드 간격 `12dp`
- 카드 높이 약 `154dp`
- 흰 카드/강조 카드 혼합

각 테이블 카드 표시:
- 테이블명
- 금액
- 경과시간
- 수용 인원
- 상태 chip

### 4.6 테이블 상태 시각 규칙
현재 구현 기준:

- `EMPTY`
  - 흰색 카드
  - 상태: `빈자리`
- `OCCUPIED`
  - 선택 시 딥그린 강조
  - 상태: `식사중`
- `BILLING`
  - 상태: `결제대기`
- `DISABLED`
  - 회색 비활성 카드
  - 상태: `사용불가`
- `MERGED`
  - 브라운 계열 카드
  - 상태: `합석됨` / `(합석)` 보조 표기

추가 강조:
- 현재 선택 / 이동 대상 / 합석 대상 / 드래그 중 카드에 따라 border 색과 두께가 달라질 수 있다.

### 4.7 테이블 인터랙션
- 테이블 탭/클릭 가능
- `EMPTY` 상태 클릭 시 해당 `food/{tableId}` 로 진입
- 길게 눌러 드래그 후 다른 테이블에 드롭하면 이동/합석 흐름으로 연결
- 드롭 후 확인 다이얼로그를 띄울 수 있다

### 4.8 우측 주문 패널
- 고정 폭 약 `360dp`
- 흰색 패널
- 내부는 상단 헤더 / 리스트 / 합계 / 하단 버튼 구성

#### 4.8.1 헤더 카드
딥그린 헤더에 다음 표시:
- 선택 테이블명
- 상태 chip
- 경과시간
- 인원

#### 4.8.2 주문 리스트
표 헤더:
- `상품명`
- `수량`
- `금액`

규칙:
- 스크롤 가능
- 빈 경우:
  - `활성 주문이 없습니다`
- 리스트 하단에는 `주문 합계` 카드가 붙는다.

#### 4.8.3 하단 액션 버튼
가로 2분할 버튼:

1. `추가 주문`
2. `결제하기`

시각 규칙:
- 두 버튼 모두 큰 높이 (`68dp`)
- `추가 주문` = 슬레이트
- `결제하기` = 딥그린

현재 연결:
- `추가 주문` → `food/{tableId}`
- `결제하기` → `payment/{tableId}`

### 4.9 대화상자
드래그/대상 선택 이후 아래 확인 다이얼로그를 사용할 수 있다.

- `이동 확인`
- `합석 확인`

본문 예:
- `T-x의 주문을 T-y로 이동할까요?`
- `T-x과 T-y를 합석할까요?`

### 4.10 금지 사항
- 우측 주문 패널을 스크롤에 묻히게 하지 않는다.
- 테이블 그리드를 리스트형 단일 컬럼으로 축소하지 않는다.
- 빈 테이블 진입 흐름을 제거하지 않는다.
- 이동/합석 시각 피드백을 제거하지 않는다.

## 5) FOOD (FoodCourtScreen)

### 5.1 목적
- 선택 테이블 기준 주문 입력
- 메뉴 카테고리/상품 선택
- 수량/단가/취소 조작
- 결제 또는 시재 관련 액션 진입

### 5.2 전체 구조
- 상단: `PosTopBar`
- 본문: 좌/우 2-pane
  - 좌 weight `1.2`
  - 우 weight `1.6`
- 페이지 배경: `PAGE_BG_MAIN (#F6F2E9)`

### 5.3 예전 스펙 대비 현재 구조 변화
현재 구현은 **화면 최하단 전폭 3버튼 액션바가 아니다.**  
대신 우측 하단에 **결제/시재 액션 클러스터**가 들어간다.

이 점은 이후 수정 시 반드시 유지한다.

### 5.4 좌측 패널 — 주문 리스트
#### 5.4.1 헤더 카드
딥그린 헤더에 다음 표시:
- 선택 테이블명 (`테이블 선택` fallback 가능)
- 경과시간
- 인원

#### 5.4.2 리스트 표 헤더
- `No.`
- `상품명`
- `단가`
- `수량`
- `금액`

#### 5.4.3 행 동작
각 행은 다음을 포함한다.
- 순번
- 상품명
- 단가
- 수량 stepper (`- / 수량 / +`)
- 금액
- 취소/복원 버튼

추가 규칙:
- 취소된 항목은 취소선 + 빨간색 표시 가능
- 금액 클릭으로 단가 변경 다이얼로그 진입 가능
- 좌측 리스트 자체는 스크롤 가능
- 상/하단 스크롤 힌트 아이콘 사용 가능

#### 5.4.4 좌측 하단 액션
현재 버튼은 2개:

1. `행사할인`
2. `전체취소`

규칙:
- `행사할인` = 슬레이트 강조
- `전체취소` = 아웃라인 + 빨간 텍스트 강조

#### 5.4.5 하단 합계 카드
딥그린 프리미엄 카드에 다음 표시:
- `총 매출`
- `할인 금액`
- `받을 금액`

강조 규칙:
- `받을 금액`은 금색 계열로 크게 표시

### 5.5 우측 패널 — 카테고리 + 메뉴 + 결제 액션
#### 5.5.1 카테고리 탭
- `ScrollableTabRow`
- 기본 첫 탭은 `즐겨찾기`
- 나머지는 매장/메뉴 카테고리
- 첫 탭은 별 아이콘 사용 가능

현재 예시 카테고리:
- `즐겨찾기`
- `오므&커리`
- `본까스`
- `해천죽`
- `이심사철기`
- `미코`

#### 5.5.2 즐겨찾기 탭 특이사항
- 즐겨찾기 탭의 첫 카드에 `추가` 카드가 존재할 수 있다.
- 이를 통해 즐겨찾기 선택 다이얼로그를 띄운다.

#### 5.5.3 상품 그리드
- 4열 그리드
- 카드 높이 약 `100dp`
- 카드에는 메뉴명 + 가격 표시
- 클릭 시 선택 테이블 주문에 상품 추가

#### 5.5.4 우측 하단 액션 클러스터
현재 구조는 `1 + 1 + 2x2` 형태다.

왼쪽 세로 2개:
- `테이블 화면 이동`
- `기타시재`

오른쪽 2x2 tender grid:
- `상품권`
- `현금`
- `H.Point 사용`
- `카드/모바일`

시각 규칙:
- `테이블 화면 이동` = 연한 웜 그레이/핑크 계열
- `기타시재` = 슬레이트
- `상품권`, `H.Point 사용` = 베이지
- `현금`, `카드/모바일` = 딥그린
- 버튼 크기는 크고, 아이콘 + 라벨 세로 배치

현재 연결:
- `테이블 화면 이동` → 이전 화면 복귀
- `현금`, `카드/모바일` → `payment/{tableId}`
- 기타 버튼은 현재 placeholder 가능

### 5.6 다이얼로그
현재 Food 화면은 다음 다이얼로그를 가질 수 있다.

- `즐겨찾기 상품 선택`
- `단가 변경`
- `전체취소 확인`

이들 다이얼로그는 UI 일부로 간주한다.

### 5.7 금지 사항
- Food를 하단 전폭 3버튼 구조로 되돌리지 않는다.
- 우측 하단 결제 클러스터를 일반 리스트로 바꾸지 않는다.
- 좌측 리스트의 수량 stepper / 취소 복원 UX를 제거하지 않는다.

## 6) PRODUCT_REGISTER

### 6.1 목적
- 상품 스캔 또는 수기 입력으로 카트에 상품 추가
- 좌측 카트 / 우측 상품 선택 구조 유지
- 판매용 등록/결제 진입 화면으로 사용

### 6.2 전체 구조
- 상단: `PosTopBar`
- 본문: 좌/우 2-pane
  - 좌 weight `0.45`
  - 우 weight `0.55`
- 페이지 배경: `PAGE_BG_MAIN (#F6F2E9)`

### 6.3 핵심 의미
이 화면은 반드시 아래 의미를 유지한다.

- 판매용 상품 입력 화면
- 카트 기반 합계 확인 화면
- 하단 tender/action 진입 화면
- 상품 마스터 등록 폼이 아님

### 6.4 좌측 패널 — 카트/합계
#### 6.4.1 메인 카트 컨테이너
- 흰색 Surface + 회색 border
- 상단 헤더 / 리스트 / 액션 row 구성

#### 6.4.2 카트 헤더
표 헤더는 현재 아래 라벨을 사용한다.

- `No`
- `상품명`
- `단가`
- `수량`
- `금액`

기존 문서의 `No.` 가 아니라 현재 소스 기준 `No` 로 사용한다.

#### 6.4.3 카트 행
각 행은 다음을 포함한다.
- 순번
- 상품명
- 2번째 줄 바코드
- 단가
- 수량 stepper
- 금액
- 삭제 아이콘

시각 규칙:
- 행 높이 약 `66dp`
- 수량 컨트롤은 양쪽 아이콘 버튼 구조
- 금액은 딥그린 강조
- 리스트 스크롤 시 상/하단 힌트 아이콘 가능

#### 6.4.4 빈 상태
카트가 비어 있으면 중앙 empty state를 사용한다.

구성:
- 쇼핑카트 아이콘
- `주문 상품이 없습니다`
- `홈으로 돌아가기` 버튼

규칙:
- 홈 버튼은 딥그린 강조
- empty state는 좌측 카트 영역 중앙에 배치

#### 6.4.5 좌측 하단 액션 row
현재 버튼은 3개:

1. `행사적용`
2. `주문 보류`
3. `전체취소`

시각 규칙:
- `행사적용`, `주문 보류` = 아웃라인 계열
- `전체취소` = 연한 붉은 배경 + 빨간 텍스트

#### 6.4.6 하단 합계 카드
딥그린 프리미엄 카드에 아래 항목을 표시한다.

- `총 매출`
- `할인 금액`
- `받을 금액`

강조 규칙:
- `받을 금액`은 가장 크게 표시
- 금색 계열 숫자 강조 사용 가능

### 6.5 우측 패널 — 카테고리 / 바코드 입력 / 상품 그리드 / 하단 액션
#### 6.5.1 카테고리 탭
현재 카테고리 목록:
- `즐겨찾기`
- `채소`
- `과일`
- `생수`
- `종량제 봉투`
- `기타`

규칙:
- `ScrollableTabRow`
- 첫 탭은 별 아이콘 사용 가능
- 선택 탭은 굵은 폰트 + 딥그린 강조

#### 6.5.2 수기 바코드 입력 줄
카테고리 탭 아래에 수기 입력 row를 둔다.

구성:
- `OutlinedTextField`
  - placeholder: `바코드 수기입력`
- `키패드` 버튼

규칙:
- 입력은 숫자만 허용
- Done 또는 키패드 OK로 스캔 처리
- 키패드 버튼은 슬레이트 계열

#### 6.5.3 상품 그리드
- 4열 그리드
- 카드 높이 약 `80dp`
- 카드 표시:
  - 상품명 (최대 2줄)
  - 가격
- 카드 클릭 시 카트에 +1

#### 6.5.4 우측 하단 tender/action 영역
현재 구조는 **2행 x 3열처럼 보이지만, 우측 열이 더 넓은 비대칭 가중치**를 가진다.

1행:
- `장바구니할인`
- `상품권`
- `현금`

2행:
- `기타시재`
- `H.Point 사용`
- `카드/모바일`

색상 규칙:
- `장바구니할인`, `기타시재` = 슬레이트
- `상품권`, `H.Point 사용` = 베이지
- `현금`, `카드/모바일` = 딥그린

크기 규칙:
- 버튼 높이 약 `84dp`
- 우측 `현금`, `카드/모바일` 이 더 넓다 (`weight 1.2`)

이 영역은 스크롤 하단에 밀려 사라지는 일반 리스트가 아니라,
**우측 pane 하단의 주요 액션 클러스터**로 유지한다.

현재 구현 기준:
- 시각/배치/UI는 완성되어 있다.
- 각 버튼의 실제 결제/정산 로직 연결은 아직 placeholder일 수 있다.
- 따라서 이 영역의 존재와 라벨은 유지하되, 동작은 추측해서 바꾸지 않는다.

### 6.6 수기 키패드 다이얼로그
`키패드` 버튼을 누르면 수기 바코드 입력용 다이얼로그를 띄운다.

구성:
- 제목: `바코드 수기입력`
- 우측 상단 닫기 아이콘
- 큰 숫자 표시 영역
- 3열 숫자 키패드
- 하단:
  - `C`
  - `0`
  - `OK`

규칙:
- `OK` 는 딥그린 강조
- `C` 는 빨간 계열
- 일반 숫자 키는 흰색 키패드

### 6.7 바코드 처리 UX
- route barcode 진입 가능
- 수기 입력 가능
- 스캐너 gateway 연결 가능
- 등록되지 않은 바코드는 스낵바 메시지
- 등록 성공 시 `OO 상품이 추가되었습니다.` 메시지

### 6.8 금지 사항
- ProductRegister를 폼형 등록 페이지로 바꾸지 않는다.
- 우측 하단 tender/action 영역을 제거하지 않는다.
- 수기 바코드 입력줄과 키패드 다이얼로그를 없애지 않는다.
- empty state의 홈 복귀 UX를 제거하지 않는다.

## 7) PAYMENT

### 7.1 목적
- 선택된 주문 금액에 대해 결제 수단별 금액을 입력/배분한다.
- 총 결제금액과 받은 금액을 비교 확인한다.

### 7.2 전체 구조
- 상단: `PosTopBar`
- 본문: 좌/우 2-pane
  - 좌 weight `1.0`
  - 우 weight `1.1`
- 페이지 배경: `PAGE_BG_MAIN (#F6F2E9)`

### 7.3 좌측 패널 — 결제 요약
흰색 카드에 다음 구성:

1. 테이블명
2. 결제수단 / 결제금액 리스트
3. 각 라인 삭제 버튼
4. 하단 요약
   - `총 결제금액`
   - `받은 금액`

강조 규칙:
- `총 결제금액` = 빨간색 강조
- `받은 금액` = 딥그린 강조

### 7.4 우측 패널 — 결제 수단 + 입력 금액 + 숫자 키패드
#### 7.4.1 결제 수단 그리드
- 3열 x 2행 그리드
- 카드 높이 약 `100dp`
- 선택 시 딥그린 배경으로 반전

현재 수단:
- `카드/모바일`
- `현금`
- `상품권`
- `H-POINT`
- `제휴카드`
- `간편결제`

주의:
- 상위 화면의 버튼 라벨은 `H.Point 사용` 이지만,
  Payment grid에서는 현재 `H-POINT` 라벨을 사용한다.

#### 7.4.2 입력 금액 카드
- `선택된 수단: ...`
- `입력 금액`
- 현재 숫자 입력값 표시

#### 7.4.3 숫자 키패드
현재 4행 구성:

1행:
- `7`, `8`, `9`, `Clear`

2행:
- `4`, `5`, `6`, `Backspace`

3행:
- `1`, `2`, `3`, `Enter`

4행:
- `0`, `00`, `만원`

시각 규칙:
- `Enter` = 딥그린
- `Clear`, `Backspace` = 슬레이트
- 숫자 키 = 흰색 + 회색 border
- `만원` 은 더 넓은 키

#### 7.4.4 하단 닫기 버튼
- 전폭 버튼
- 라벨: `결제 화면 닫기`
- 슬레이트 배경
- 현재 동작은 `popBackStack()` 기반 복귀

### 7.5 금지 사항
- Payment를 단순 단일 키패드 팝업으로 축소하지 않는다.
- 결제 수단 그리드를 리스트나 드롭다운으로 바꾸지 않는다.
- 좌측 결제 요약 패널을 제거하지 않는다.

## 10) 유지보수 가드레일

### 10.1 반드시 유지할 것
- 공통 `PosTopBar`
- 2-pane 중심 POS 레이아웃
- 딥그린 / 베이지 / 슬레이트 색 분리
- 큰 버튼 / 큰 숫자 / 큰 금액 강조
- 결제/시재 액션 영역의 고정적 성격
- ProductRegister의 판매 등록 의미
- Food의 우측 하단 액션 클러스터 구조
- Restaurant의 고정 우측 주문 패널
- Payment의 3x2 결제 수단 그리드 + 숫자 키패드

### 10.2 바꾸면 안 되는 것
- ProductRegister를 관리용 상품등록 폼으로 변형
- Food를 예전 3버튼 전폭 하단바 구조로 되돌림
- SuperHome Hero 타일 의미 변경
- Restaurant의 빈 테이블 → Food 진입 제거
- Payment의 전용 결제 수단 화면 구조 제거

### 10.3 변경 시 원칙
- 기존 UI 컨셉을 유지한 상태에서 확장한다.
- 관련 화면만 최소 범위로 수정한다.
- 새로운 버튼/카드/패널을 추가할 때도 현재 디자인 언어 안에서 해결한다.
- 비대칭 가중치(예: 현금/카드 버튼 확장) 같은 현재 레이아웃 의도를 임의로 평준화하지 않는다.

## 11) 한 줄 요약

현재 UI는  
**“THE HYUNDAI 오프라인 POS용 베이지-화이트 기반 2-pane 대형 터치 UI”** 이며,  
핵심 특성은 다음 5가지다.

1. 상단 고정 브랜드 탑바
2. 화면별 2-pane 구조
3. 딥그린 / 베이지 / 슬레이트 색 역할 분리
4. 결제/시재 액션의 크고 명확한 클러스터형 배치
5. ProductRegister / Food / Payment / Restaurant 간 흐름이 끊기지 않는 POS형 조작 UX
